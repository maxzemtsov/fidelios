# v0.0.47

> Released: 2026-05-17

The in-app file viewer now finds files anywhere in the repo, opens them on the host, and links straight to the file on GitHub.

## Fix: files outside the workspace cwd were "not found"

A workspace `cwd` can point at a *subdirectory* of a git repo (e.g. `…/TraitTune_v2/marketing`). The file viewer searched only inside that `cwd`, so a referenced file living in a sibling tree (`…/TraitTune_v2/docs/marketing/exhibits/COMPETITOR_SCAN.md`) reported "File not found" even though it was on disk.

The viewer now resolves the **git repo root** from the workspace `cwd` (`git rev-parse --show-toplevel`) and searches from there. A file anywhere in the repo resolves — including uncommitted-but-saved files, which `git ls-files --others` already lists, so a freshly written file opens *before* any commit.

## Reveal a workspace file on the host

FideliOS runs on the host where the workspace files live, so a file no longer has to be previewed or downloaded to be opened. The viewer now has a **Reveal in Finder** button: the server runs `open -R` on the resolved file, selecting it in Finder so you can open it however you like — including the OS "Open With" picker.

Available for any resolved file (markdown, text, PDF, Excel, …), committed or not. Gated to `local_trusted` deployments, so a remote instance never opens files on a server.

## Deep file links instead of a bare repo link

When a file genuinely is not on disk, the fallback used to link to the *whole repository*. It now builds a deep link to the file itself — `…/blob/<branch>/<path>`, derived from the workspace's `repoUrl` and `repoRef` (ssh remotes are normalized to https). The viewer shows **Open file on GitHub**, falling back to **Open repository** only when no path-level link can be built.
