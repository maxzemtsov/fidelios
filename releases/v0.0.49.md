# v0.0.49

> Released: 2026-05-18

The first two pillars of safe parallel multi-agent development: blocker dependencies are now enforced, and agent instructions adopt a one-issue-one-branch-one-PR Git Workflow.

## Blocker enforcement

[#80](https://github.com/fideliosai/fidelios/pull/80) — `blocked_by` issue relations were recorded but never enforced; an agent could check out and start an issue whose dependency was still open.

- `checkout()` rejects (409) a fresh checkout of an issue with an unresolved `blocked_by` relation; the conflict carries the blocking issues. A re-checkout by the agent already running the issue is exempt, so a blocker added mid-run never interrupts an in-flight run.
- The agent inbox (`/agents/me/inbox-lite`) skips issues with unresolved blockers, so blocked work never reaches an agent's queue.
- Dependency wakeup — waking a blocked issue's assignee when its last blocker resolves — already existed and is unchanged.

## Canonical Git Workflow

[#81](https://github.com/fideliosai/fidelios/pull/81) — agent instructions told agents to share one "root parent branch" across every sub-issue of an issue tree, which puts multiple agents on a single branch — a primary cause of parallel-work collisions. Replaced with **one issue → one branch → one PR**:

- `AGENTS.md` §11 (the repo's own contributor workflow), the `onboarding-assets/default` instruction templates, and the `fidelios-create-agent` skill all adopt the per-issue branch + PR + merge-queue model.
- The `fidelios-create-agent` skill now requires the CEO to author a Git Workflow section into each hire's `HEARTBEAT.md`, naming the repo's real integration trunk.
