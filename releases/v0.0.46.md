# v0.0.46

> Released: 2026-05-17

A reliability pair: the in-app file viewer now resolves files on large repos, and runs killed by an external signal are classified honestly.

## Fix: file viewer fails to find files on large repos

[#74](https://github.com/fideliosai/fidelios/pull/74) — clicking a document reference in a comment (the v0.0.44 viewer) showed "File not found in this issue's workspace" even when the file was present. The workspace search walked the filesystem with a 20,000-entry cap; on a large repo (~17.7k files) it bailed before reaching the target, and directory scan order is non-deterministic.

- Search now uses `git ls-files` (tracked + untracked-non-ignored files): instant, uncapped, and `node_modules`/build dirs are excluded for free. It matches by exact path, path suffix, or basename. The bounded filesystem walk remains as a fallback for non-git workspaces.
- A file that genuinely cannot be resolved no longer dead-ends. The endpoint returns `kind: "missing"` (HTTP 200) carrying the searched workspace directory and the project's repository URL; the viewer shows a clear message with an **Open repository** link.
- Binary files (PDF, Excel, images) continue to download to the host — `res.download()` sends `Content-Disposition: attachment`, so the browser saves the file and the host opens it with its native app rather than rendering it inline.

## Fix: runs terminated by a signal mislabeled `adapter_failed` (FID-54)

[FID-54](https://github.com/fideliosai/fidelios/pull/75) — when the `claude` process was killed by an external signal (a SIGTERM from a watchdog or the OS, surfacing as exit code 143/137/130), the claude-local adapter produced no result JSON and the run fell through to a generic `adapter_failed`. That label is misleading: the adapter did not fail — its child was terminated.

- A new `isSignalTerminated` check treats `SIGTERM`/`SIGKILL`/`SIGINT` or exit code `143`/`137`/`130` as a termination, not an adapter fault.
- Such a run is now classified `timeout`, both when no result JSON was produced and when a partial result was parsed. A genuine `claude_auth_required` (v0.0.43) still takes precedence, and a `subtype: success` run is still never failed.
