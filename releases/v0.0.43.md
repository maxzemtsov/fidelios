# v0.0.43

> Released: 2026-05-17

A targeted reliability fix for the Claude (local) adapter: agents no longer fail spuriously with `claude_auth_required`.

## Fix: spurious `claude_auth_required` failure cascade

[FID-56](https://github.com/fideliosai/fidelios/pull/70) — claude-local agents whose work touched authentication or login (a `/login` route, an auth form, debugging Claude auth itself) repeatedly failed with `Claude run failed: ... Not logged in · Please run /login (claude_auth_required)`, and the heartbeat retried the mislabeled runs in an endless ~30s loop.

Root cause: `detectClaudeLoginRequired` regex-scanned the model's *own output* (`parsed.result` and the full `stdout` stream) for auth phrases, and `execute.ts` stamped the run `claude_auth_required` with no check that the run had actually failed. A successful run (`subtype: success`) that merely *mentioned* `/login` flagged itself as an authentication failure.

- `detectClaudeLoginRequired` now reads auth state only from the CLI's error channels: `stderr` and structured `errors[]` always; the `result` text only on a genuinely failed result (`subtype !== "success"`); `stdout` only when no result JSON was produced.
- `execute.ts` gates the `claude_auth_required` code on a non-success result — a `subtype: success` run is never an auth failure.
- The auth-detection regex is tightened to CLI-specific phrasings (`not logged in`, `invalid api key`, `please run /login`, …).
- Stale `claude login` guidance corrected to `claude auth login` in the environment probe, quota error messages, and the onboarding / agent-detail UI.
- New `claude-local-auth-classifier.test.ts` — 5 unit tests + 2 `execute()` smoke tests.

This is a follow-up to FID-56 PRs #68 and #69, which corrected the login command and the retry classification but left the misclassification root cause in place.
