# v0.0.52

> Released: 2026-05-19

Four changes — three adopted from a fresh comparison against the
`paperclipai/paperclip` upstream (optimistic locking for agent instruction
files, a configuration-diagnostic skill, a blocked-work inbox) and one new
FideliOS capability: native per-company merge coordination.

## Doc locking for instruction bundles

[#88](https://github.com/fideliosai/fidelios/pull/88) — concurrent edits to an
agent's instruction bundle (`AGENTS.md`, `SOUL.md`, …) went through a bare
`fs.writeFile` with no guard, so two editors saving the same file silently
clobbered each other.

- Reads now return a content-hash `etag`, and
  `PUT /agents/:id/instructions-bundle/file` accepts an optional `baseEtag`. A
  stale `baseEtag` is rejected with HTTP 409 (`Agent instructions file was
  updated by someone else`) instead of overwriting.
- The agent instruction editor round-trips the etag and, on a 409, shows a
  conflict banner with overwrite / discard-and-reload actions.

## Diagnose agent config skill

[#89](https://github.com/fideliosai/fidelios/pull/89) — a new
`diagnose-agent-config` skill inspects an agent (or every agent in a company)
and reports configuration-health problems with a recommended fix for each: a
relative or temp-directory `cwd`, instruction-bundle warnings, an empty or
non-editable bundle, a project workspace with no repo remote, a missing model.
It is read-only — it reports and never edits configuration.

## Blocked inbox

[#90](https://github.com/fideliosai/fidelios/pull/90) — blocked work was buried
in the normal issue list.

- A new `attention=blocked` filter on the issues list returns blocked issues —
  those explicitly `blocked` and those with an unresolved `blocked_by`
  dependency — each annotated with why it is blocked: `manually_blocked`,
  `blocked_by_dependency`, or `stale_dependency` (every tracked blocker
  resolved).
- A standalone **Blocked** page (sidebar → Work → Blocked) groups them by
  reason so the ready-to-unblock issues stand out.

## Per-company merge slot

[#92](https://github.com/fideliosai/fidelios/pull/92) — parallel engineer
agents in a company could merge into the trunk at the same time; two PRs each
CI-green against an *older* trunk can land together and break it. GitHub's
merge queue solves this but requires an Enterprise plan on private repos. This
is the FideliOS-native equivalent — free, and uniform across every company.

- A new `merge_locks` table grants one **merge slot** per company, enforced by
  a partial unique index. The service mirrors the issue checkout-lock: a
  conditional `INSERT ... ON CONFLICT DO NOTHING` is the whole concurrency
  primitive.
- `POST /api/companies/:companyId/merge-lock` acquires (or renews) the slot —
  non-blocking, returning `{"acquired":true}` or `{"acquired":false,"heldBy":…}`
  so the agent polls rather than failing. `DELETE` releases it; `GET` reports
  status.
- A stale slot — its 30-minute TTL expired, or its holder run terminal — is
  reclaimed on contention and by a reaper on the existing scheduler tick, so a
  crashed agent never blocks a company's merges.
- Engineer agents acquire the slot before `gh pr merge` and release after; the
  workflow is documented in the engineer `HEARTBEAT.md` "Merge Slot" section,
  repo `AGENTS.md` §11, and the fidelios skill.
