# v0.0.37

> Released: 2026-04-21
> Windows fix: local CLI adapters no longer crash with EPERM when the user doesn't have Developer Mode enabled.

## What's fixed

Right after v0.0.36 fixed the CMD path-quoting bug, the very first real
Codex agent assignment on Windows failed with:

```
EPERM: operation not permitted,
symlink 'C:\Users\Sergey F\.codex\auth.json'
     -> 'C:\Users\Sergey F\.fidelios\instances\default\companies\<id>\codex-home\auth.json'
```

Windows blocks `fs.symlink` for unprivileged users unless **Developer
Mode** is enabled — which is an unreasonable ask for our non-technical
target audience. FideliOS was using raw symlinks in two places:

1. **Codex `auth.json` seeding** — the worktree-isolated codex-home
   per company pulled in the user's main auth via symlink
2. **Skills directory injection** — every local CLI adapter (Codex,
   Claude, Cursor, Gemini, Pi, Opencode) symlinked FideliOS skills
   into the CLI's global skills dir

Both paths now use a layered fallback that lands on something
unprivileged Windows users can actually do:

| Kind | Chain |
|------|-------|
| Files (auth.json) | `symlink` → `hardlink` → `copy` |
| Directories (skills) | `symlink` → `junction` (Windows) → recursive copy |

Junctions and NTFS hardlinks work without admin rights for local
paths. Copy is last resort and FideliOS logs a hint when it lands
there so you know to re-run the agent if the source file changes
(relevant when Codex rotates the auth token).

## How to pick up the fix

On Windows:

```powershell
iwr -useb https://fidelios.nl/install.ps1 | iex
```

or just upgrade the CLI directly:

```
npm install -g fidelios@latest
```

Then reassign the Codex agent — the symlink error is gone and runs
complete normally.

## Scope

Applies to every local CLI adapter:

- `codex_local` — auth.json seeding + skills injection
- `claude_local` — skills injection
- `cursor_local` — skills injection
- `gemini_local` — skills injection
- `pi_local` — skills injection
- `opencode_local` — skills injection
